Module 1: Discuss and capture Backlog

This module will provide an understanding of how OKRs translate to a Backlog, what format should a User Story have, what is the Acceptance criteria and finally, how to estimate and breakdown tasks. The main goals are:

  1. Come to consensus on the definition of Epic, Feature, and User Story (EFUs) and how it aligns with Objectives and Key Results (OKRs)
  2. Establish understanding of what a "good" User Story is using the INVEST framework
  3. Practice writing "good" Acceptance Criteria
  4. Establish understanding of how Estimates, Targets, and Commitments align
  5. Practice Refinement from Agile ceremonies
  6. (Stretch Goal) Establish understanding of how to initiate a development sprint with a Sprint Commitment and Task Breakdown

Prerequisites - Module 1

  1. Divide the teams into separate breakouts
  2. Present the POV slide titled "CEO's OKRs in Diagram"
  3. Coaches will guide the team in one of two (hopefully both) directions:
    • Elucidating backlog items from Objectives and Key Results
    • Executing Backlog Refinement and Sprint Planning ceremonies

Exercise 1: Translating Objectives and Key Results into Epics, Features, and User Stories

After this exercise, class participants will have an understanding of how to translate business objectives into actionable development work that can be prioritized using feasibility, desirability, and viability.

  1. Reiterate the Mission and Vision from the Role Play exercise.

  2. Decide for the class that they will focus on the "Grow our business" objective and its KRs.

  3. Reiterate that SmartHotel360 can provide the outcomes needed to achieve Key Results.

  4. Start group discussion on how to encourage users to make more online coupons.

    • Some teams will come up with a lot of ideas such as loyalty programs, B2B programs, etc. It's the coaches job to decide if a team is mature enough to pursue their own initiatives while maintaining the option to corral the discussion in the direction of the prescribed Lab scope.

  5. Guide the class into translating these business ideas into releasable Features.

  6. Guide the class into grouping those Features into Epics.

  7. Move onto the User Story Format exercise

User Story Format

After this exercise, class participants will understand the format of a User Story, how to write a User Story guided by the "INVEST" principles, and how to identify Acceptance Criteria.

You have the option to guide the class, and provide structure around a single Feature, "Add Discounts to Reservations from the hotel's website", from a "Entice Customers" Epic.

  1. Divide the team into two groups. One group will represent the Hotel Guest persona, the other will assume the Hotel Owner persona.

  2. (5 minute team exercise) Each group will discuss one or two User Stories that align with the Redeem Coupons feature. They need to:

    • On the whiteboard, write out the INVEST guidelines:

      • “I” ndependent (of all others)
      • “N” egotiable (not a specific contract for features)
      • “V” aluable (or http://guide.agilealliance.org/guide/incremental.html)
      • “E” stimable (to a good approximation)
      • “S” mall (so as to fit within an iteration)
      • “T” estable (in principle, even if there isn’t a test for it yet)

    • On the whiteboard, write out the User Story in the correct format

    • On the whiteboard, provide a Description for that story that describes the business value that needs to be delivered from that story.

  3. Each group shares their User Story and Description and elicits feedback from the class.

  4. Each group will discuss and document the Acceptance Criteria for their Story(s)

    • On the whiteboard, describe the Acceptance Criteria in a bulleted list. (Remember that Acceptance Criteria should help the team test for business value and completeness!)

  5. Each group shares their Acceptance Criteria and elicits feedback from the group.

Some good User Story examples that follow the "As a Persona, I want to Action, so that Value" guidance and their acceptance criteria.

The below example follow this pattern:

Persona

  • User Story
    • Acceptance Criteria

Hotel Guest

  • As a Hotel Guest, I want to use the hotel's website to redeem coupon codes I received in an e-mail promo
    • From the reservation management page of the hotel's website users should be able to apply a coupon code to their existing reservation.
  • As a Hotel Guest, I want to add a coupon code to my cart so that I can save money on booking
    • From the e-shopping cart, in the Billing portion of the form, users should be able to apply a coupon code and see the updated price reflected in the shopping cart.
  • As a Hotel Guest, I want to use the hotel's website to redeem coupon codes I received in an e-mail promo
    • Users can only submit valid, non-expired, Coupon Codes
    • Users can only submit a single Coupon Code per reservation
    • The booking price should be automatically updated when a Coupon Code is applied
    • The redeemed coupon code should be always visible on the reservation page after it has been applied

Hotel Owner

  • As a Hotel Owner, I want to provide customers with coupon codes so that I can attract new customers
    • From the reservation management screen, I should be able to apply a discount to a customer's existing reservation in a dollar amount or percentage.
  • As a Hotel Owner, I want to provide discounts to customers' reservations so that I can retain and delight loyal customers
    • From the management dashboard, provide an e-mail address, discount amount, and a unique discount code, and be able to press a single button to send to a single, or list, of email addresses.
  • As a Hotel Owner, I want to provide discounts to customers' reservations so that I can retain and delight loyal customers
    • Hotel Owners can only apply valid, non-expired, Coupon Codes
    • Hotel Owners may apply more than one Coupon Code
    • Hotel Owners may choose to provide a simple dollar amount discount in whole dollars and cents ($###.##)
    • Hotel Owners may choose to provide no more than a 50% discount on any existing reservation
    • The booking price in the existing reservation should be automatically updated when a discount is applied

Estimation

If you need to, you can guide the class to shape the discussion around a single User Story - "As a Hotel Owner, I want to provide discounts to customers' reservations so that I can retain and delight loyal customers, from the Feature - "Add Discounts to Reservations from the hotel's website".

  • Perform Planning Poker on the User Story "As a Hotel Owner, I want to provide discounts to customers' reservations so that I can retain and delight loyal customers"
  • Perform Task Breakdown of the User Story "As a Hotel Owner, I want to provide discounts to customers' reservations so that I can retain and delight loyal customers"
Note

Be sure to explicitly explain the difference in estimation around complexity for User Story Points and hours remaining for Tasks.
