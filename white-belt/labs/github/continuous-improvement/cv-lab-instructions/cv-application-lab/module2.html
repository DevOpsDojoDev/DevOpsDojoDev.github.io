Module 2: GitHub KPI Dashboard Walkthrough

You will be able to see 6 metrics in the overview.

The first one is Pull Request, which GitHub Insights is monitoring. The next four key metrics are identified by DevOps Research and Assessment (DORA) team that indicate the performance of a software development team. These metrics are Lead Time, Change Failure Rate, Deployment Frequency and Mean Time to Restore. The last one is a newly added metric around Availability and this is being worked on.

Since the data from GitHub may be insufficient at some moments so for the demonstration purpose, we introduce the Test Data Management for a better visualization and you can switch between two data sources.

Exercise 1: KPI Introduction

You can go deep with each metric and select the organization, repository, pipelines and data range to see the exact data. The dashboard view may be slightly different due to continuous adjustment.

Lets get started with Pull Request Analysis includes the PR Number, PR Size, PR Time and PR Quality. It also provides insights into Average Code Review Turn Around time and Average open time for a PR. It’s more at the level of the code itself.

Deployment Frequency shows how often an organization successfully releases to production. We are capturing this metrics for dev, qa as well as prod environment. You can find daily average, number of deployments and deployment trend here.

Lead Time for Changes is the amount of time it takes a commit to get into production. We can find the number of features in terms of New, active as well as closed features . It also shows Feature Trend in terms of Active and closed featured in selected duration as well as Lead time to change trend in the period.

Change Failure Rate is the percentage of deployments causing a failure in production. We can see the distribution by failure reasons, Failure Trend for different stages, average change failure rate and change failure rate in production .

Mean Time to Restore shows how long it takes to recover from a failure in production. We can find the number of issues in terms of new, active, verified, resolved and closed issues. The issue Trend and MTTR trend over selected period. Also we can see the time trend in terms of how much time it takes for a issue to move from one state to another for example moving from New state to Active.

Exercise 2: High Level Architecture

What happens behind the scene?

Logic apps will retrieve the raw data of specific organizations and repositories from GitHub API through personal access token. Following data model, the data will be stored in the staging database. Then Data Factory will integrate these data based on metrics measured and store them in the core database, where you can also find the test data generated by TDM. They are separated by the data source column. Connected to the SQL server, Grafana dashboard now can do real-time query and digest data from the core database, showing those key metrics.
