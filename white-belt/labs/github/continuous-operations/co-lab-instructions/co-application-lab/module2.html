Module 2: Simulate SLO metrics pass/fail scenario

In this module, we will see 2 scenarios, firstly the PASS scenario in which the availability and latency are below the threshold limits, and the application is released to PRD.

Exercise 1: Check passed Latency and Availability metrics in QA environment

In this exercise, we will walk-through the success scenario of executed CD workflow in QA stage.

Prerequisites - Exercise 1

  • Module 1 to configure code changes for CD workflow should be completed.
  • Pull request should be merged and deployment to dev and qa stages should have been completed.

  1. Go to last executed CD workflow in QA environment from actions. Expand the logs for check latency. The latency value is below the threshold which is set at 20000, hence the latency SLO evaluation has been passed as shown below. This implies that for all the transactions that took place on PRD application, the latency was less than the threshold and thus the application is running at optimal levels in production environment.

  2. Expand and check the logs for passed check availability. The value in output is '1' which means Coupons production site is available. This check helps us ensure that before releasing to production, the production app service is up and running.

Exercise 2: Simulate failed availability metric in QA environment

This is an optional exercise. In this exercise, we will simulate the failed scenario of executed CD workflow in QA stage.

Prerequisites - Exercise 2

  • Check with coach if context setting is done for the failed scenario for SLO metrics. For the failed scenario, stop the App Service so that the App Insights tracks it as unavailable.
  • Module 1 to configure code changes for Coupons YAML workflow should be completed.
  • Pull request should be merged and deployment to dev and qa stage should have been completed.
  • Exercise 1 should be completed.

  1. Go to last executed CD workflow in QA environment from actions. Expand the logs for check latency. The latency value is below the threshold which is set at 20000, hence the latency SLO evaluation has been passed as shown below. This implies that for all the transactions that took place on PRD application, the latency was less than the threshold and thus the application is running at optimal levels in production environment.

  2. Post completion of execution, check qa stage and look at the summary for the executed jobs.

  3. Click on SLO_CHeck and expand the logs for check availability. The value in output is '0' which means production site is not available.

  4. Based on failed SLO check for availability and passed SLO check for latency, one of the two SLO metrics evaluation has been failed, the workflow has been failed and release to production is stopped.

Key takeaways

  • Understand failed scenario for SLO metrics.
  • Walk-through of CD workflow for stopping release of feature to production env based on evaluation of predefined SLO checks.
